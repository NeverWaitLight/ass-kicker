## Context

当前系统缺乏统一的消息发送机制，不同的消息渠道（如邮件、短信）各自实现发送逻辑，导致代码重复且难以维护。为了支持多种消息发送方式并便于未来扩展，需要设计一个抽象的Sender接口。

## Goals / Non-Goals

**Goals:**
- 设计统一的Sender抽象接口，支持多种消息发送方式
- 实现阿里云邮箱发送功能作为首个具体实现
- 提供清晰的扩展机制，便于后续添加其他发送方式（如短信、微信通知等）
- 确保配置灵活性，支持不同渠道的特定配置

**Non-Goals:**
- 实现所有可能的消息发送方式
- 修改现有业务逻辑以使用新的Sender接口（此为后续任务）
- 处理消息发送失败后的复杂重试逻辑

## Decisions

1. **采用抽象接口而非抽象类**：使用接口可以提供更大的灵活性，允许具体实现类继承其他类。

2. **统一的消息发送方法签名**：定义统一的入参和返回对象，确保所有实现类具有一致的调用方式。

3. **基于Spring Boot的配置机制**：利用Spring Boot的@ConfigurationProperties机制处理不同渠道的特定配置。

4. **依赖注入管理实例**：通过Spring容器管理不同Sender实例的生命周期，便于管理和测试。

## Risks / Trade-offs

[Risk] 不同消息渠道的特性差异可能导致统一接口设计过于复杂 -> 在接口设计时保留足够的扩展字段和方法，允许具体实现处理特殊需求

[Risk] 配置管理可能变得复杂 -> 使用清晰的配置命名约定，按渠道分组配置项

[Risk] 消息发送性能问题 -> 在具体实现中考虑异步发送机制，避免阻塞主业务流程
