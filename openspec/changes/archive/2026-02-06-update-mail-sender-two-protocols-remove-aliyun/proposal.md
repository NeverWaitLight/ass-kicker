## Why

当前系统邮件发送能力只绑定了单一阿里云邮件 sender 且依赖其特有配置无法同时支持不同邮件服务协议在多云或迁移场景下扩展性差并且增加了对单一厂商的耦合度需要通过统一抽象来支持多种协议并移除对阿里云 sender 的硬编码

## What Changes

- 引入通用邮件发送抽象 sender 直接支持两种主流邮件协议例如基于 smtp 的协议和基于 http api 的协议并以协议为维度进行配置与选择
- 为上述两种邮件协议分别新增独立的配置项包括服务器地址端口认证方式安全策略超时重试等并通过统一配置结构进行装配
- 在业务代码中将现有邮件发送调用从阿里云专用 sender 切换为新的多协议 sender 接口通过配置选择具体协议实现而不是依赖具体厂商 sdk
- 从代码和配置中彻底移除现有阿里云邮件 sender 及其专用配置字段如果有云厂商绑定逻辑统一迁移到协议级能力之上
- 对运维配置和环境变量进行整理仅保留与协议相关的通用字段清理阿里云特有字段并更新相关文档

## Capabilities

### New Capabilities

- `multi-protocol-mail-sender` 负责提供统一的邮件发送抽象并在运行时基于配置选择具体协议实现例如 smtp 协议实现或 http api 协议实现
- `mail-protocol-config` 负责定义两种邮件协议对应的配置模型加载方式校验规则以及与现有全局配置中心的集成保证不同环境下可以独立配置

### Modified Capabilities

- `mail-delivery` 在保持对外业务行为和接口不变的前提下将底层邮件发送实现从阿里云专用 sender 迁移到新的多协议 sender 以便后续可以透明切换或扩展更多协议

## Impact

- 涉及当前邮件发送相关的领域服务基础设施层邮件网关以及任何直接依赖阿里云邮件 sender 或其配置结构的代码模块
- 影响配置中心环境变量和部署配置需要新增两种邮件协议的配置项并删除阿里云邮件相关字段
- 如果存在对阿里云邮件服务 sdk 的直接依赖需要在依赖管理中移除相关库或将其退化为可选依赖
- 回归测试需要覆盖不同协议配置下的邮件发送流程包括连接失败认证失败超时重试等场景以确保新抽象不引入行为回归
