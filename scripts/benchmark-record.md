### 结论复盘：为什么优化后的效果基本可以忽略

#### 先明确测试范围

- 本次压测目标是 `/api/send` 接口的硬抗请求能力上限
- `/api/send` 走的是同步发送链路，HTTP 响应要等一次完整发送流程执行完才返回
- MQ 不参与本次压测口径，`/api/submit` 才是提交到 Kafka 的异步接口

#### 当前统计口径意味着什么

- 压测脚本已经校准：超过 deadline 才完成的成功请求，不计入 TPS
- 因此当前 TPS 代表的是“在限定时间内真正完成同步发送的请求数”
- 如果某个优化只是减少了后台收尾压力，或者只是让一部分请求稍晚一点完成，而没有把更多请求拉回 deadline 内，那么 TPS 提升就会非常有限

#### 数据说明这次优化没有打中主瓶颈

- 1000 并发：`2619.1 -> 2654.8`，提升约 `1.36%`
- 2000 并发：`2439.1 -> 2380.6`，下降约 `2.40%`
- 3000 并发：`2280.2 -> 2421.4`，提升约 `6.19%`
- 4000 并发：`2195.2 -> 2304.1`，提升约 `4.96%`
- 整体平均提升约 `2.5%`

这个幅度说明优化并非无效，而是收益只停留在边际改善，没有解除决定 `/api/send` 吞吐上限的核心瓶颈。

#### 更合理的原因判断

- 这次减少的是中间过程日志和部分 MongoDB 写入次数
- 但 `/api/send` 的关键路径仍然包含模板查询与渲染、渠道选择、渠道发送本身，以及同步链路中的线程调度与排队
- 这些成本依然要在 HTTP 返回前完成，所以它们比被删掉的日志更可能主导响应时间
- 现有优化更像是在降低后台审计与日志压力，而不是显著缩短同步发送主链路

#### 为什么会出现“看起来优化了很多，但 TPS 几乎没涨”

- 原先怀疑的“日志 I/O + MongoDB 3 次操作”不是主瓶颈，只是次要开销
- 在高并发下，真正限制吞吐的仍然是同步发送流程本身，而不是那几条被删掉的日志
- deadline 口径会放大这个事实：只有在时间窗口内完成的请求才算成功吞吐，晚完成的不再被算进 TPS
- 所以这次优化即使让系统更轻一些，也不足以把大量边缘请求从“超时后完成”拉回“deadline 前完成”

#### 本轮测试的最终结论

- 对 `/api/send` 这个同步接口来说，减少日志和减少 MongoDB 操作只带来了轻微收益
- 这说明当前 `/api/send` 的 TPS 极限主要不是被日志写入拖住，而是被同步发送主链路拖住
- 如果后续还要继续提升 `/api/send` 的极限，优化方向应优先放在同步主链路本身，而不是继续压缩日志次数

### 全链路代码分析：接下来应该优化哪里

#### 当前 `/api/send` 的真实执行链路

- 鉴权通过后进入 `SendHandler.send`
- `SendHandler` 在 `boundedElastic` 上同步执行 `sendTaskConsumer.processTask(task)`
- `processTask` 内依次执行：模板查询与渲染、渠道选择、创建发送通道、实际发送、异步审计落库
- HTTP 响应必须等上述同步主链路走完才会返回

#### 当前第一瓶颈不是日志，也不是 MongoDB，而是 `debug send delay + boundedElastic`

- `application.yml` 中当前配置了：
  - `ass-kicker.channel.debug.enabled: true`
  - `ass-kicker.channel.debug.sleep-ms: 100`
- 这意味着压测时如果没有关闭 debug 模式，每次发送都会先固定 `sleep 100ms`
- 这段休眠发生在同步发送主链路中，会直接占住执行线程
- `/api/send` 当前又运行在 `Schedulers.boundedElastic()` 上，因此线程池容量会直接决定 TPS 上限

按照当前机器核数和 `boundedElastic` 的默认行为估算，当前实测大约 `3000 TPS` 的平台，与“线程池并发上限 + 每次固定阻塞 100ms”是吻合的。  
因此当前天花板本质上是线程模型和模拟发送延迟共同形成的，不是日志 I/O 先顶满了系统。

#### 为什么前面的优化几乎没效果

- 前面的优化主要减少了中间日志和部分 MongoDB 写入次数
- 但这部分并没有移除同步主链路中的固定阻塞时间
- 只要每个请求在发送阶段仍然要阻塞约 `100ms`，而且阻塞在线程池上，那么系统吞吐上限就基本已经被锁死
- 这时减少日志和减少 MongoDB 只能降低边际损耗，无法突破主瓶颈

#### 代码层面的具体问题

##### 1. `/api/send` 走同步阻塞模型，执行器选型不适合当前压力模型

- `SendHandler` 直接 `Mono.fromRunnable(() -> sendTaskConsumer.processTask(task))`
- 然后交给 `Schedulers.boundedElastic()`
- 这意味着每个请求都会长时间占住一个 worker，直到发送完成才释放
- 当并发很高且每次发送存在固定阻塞时，线程池很快就会成为 TPS 上限

##### 2. 当前压测并不是“纯后端处理极限”，而是“后端 + 固定 100ms 模拟发送”的极限

- `ChannelDebugSimulator` 开启时，每次发送都会固定休眠
- 所以当前测试结果包含了人为注入的发送耗时
- 如果目标是测服务端自身的处理能力，这个固定 sleep 会显著扭曲结论

##### 3. 审计落库虽然已经异步，但仍然是每请求一次 MongoDB 写入

- `saveFinalRecordAsync` 里仍然是每个请求异步保存一条 `SendRecord`
- 这会继续消耗线程、Mongo 连接和序列化开销
- 在当前第一瓶颈解除后，它很可能会成为下一层瓶颈

##### 4. 同步主链路里还有重复工作

- `TemplateManager.fill` 会查模板并渲染内容
- `ChannelManager.selectChannel` 又会按模板编码再查一次模板
- `createChannel` 每次都会重新解析 `propertiesJson` 并创建通道对象
- 这些操作单次不一定很重，但在高并发下会持续累积

##### 5. 如果未来关闭 debug 改压真实通道，很多实现还存在“每请求重建客户端”的问题

- SMTP 每次创建 `JavaMailSender`
- 阿里云短信每次创建 SDK `Client`
- APNs 每次创建 `HttpClient`、读取私钥、重新签 JWT
- FCM 每次重新获取 access token

这些问题在当前 debug 模式下被 `sleep 100ms` 掩盖了，但一旦切到真实发送或移除 debug 延迟，就会很快暴露出来。

### 接下来应该优先优化什么

#### 第一优先级：优化 `/api/send` 的执行模型

- 不要继续把 `/api/send` 的完整同步发送流程绑定在 `boundedElastic` 上
- 更合理的方式是给 `/api/send` 单独使用虚拟线程执行器，或专用阻塞线程池
- 这是当前最可能直接抬升 TPS 上限的优化

#### 第二优先级：如果目标是测“后端自身硬抗极限”，必须关闭或下调 debug sleep

- benchmark 场景下建议单独使用 profile
- 至少应把 `ass-kicker.channel.debug.sleep-ms` 调成 `0`
- 否则测到的始终是“带固定发送延迟的同步接口上限”，不是服务本体的处理上限

#### 第三优先级：降低审计落库对压测结果的干扰

- benchmark 模式下可考虑直接关闭 `SendRecord` 落库
- 或改成内存缓冲后批量写入 MongoDB
- 这样更容易测出 `/api/send` 主链路本身的极限

#### 第四优先级：消除同步主链路里的重复工作

- 缓存编译后的模板，而不是每次重新编译 Mustache
- 避免同一请求中重复按 `templateCode` 查模板
- 复用已解析的 channel properties，避免每次重新反序列化
- 预构建 `ChannelConfig` 或发送器实例，减少对象创建成本

#### 第五优先级：为真实通道做连接和客户端复用

- 复用 SMTP sender、短信 SDK client、APNs HttpClient、FCM token
- 避免每请求重新创建重量级对象
- 这部分对真实发送场景影响会非常大

### 原因归纳

- 当前 `/api/send` 的吞吐上限，主要由同步阻塞发送链路决定
- 在当前配置下，这个上限首先被 `debug sleep 100ms + boundedElastic` 卡住
- 因此日志和 MongoDB 只是次要损耗项，不是决定性瓶颈
- 如果不先处理执行模型和模拟延迟，继续压日志或压 MongoDB，收益仍然只会是边际改善
